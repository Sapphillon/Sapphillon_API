# AGENTS.md

Guidance for AI agents (OpenCode / Copilot / Claude Code) working in this repository.

## What this repo is

**Sapphillon_API is a Protobuf API definition repository** — the source of truth for the Sapphillon API surface (gRPC / Connect). It is *not* a runnable server or application.

- `proto/` is the product. Edits here define the API for all downstream clients and servers.
- The Docusaurus site in `docs/` renders those proto definitions as a documentation website, deployed to GitHub Pages.
- There is no application source, server binary, or test suite in this repo.

## Proto source layout

All definitions live under `proto/sapphillon/` and are split into three domains:

| Package | Domain |
|---|---|
| `sapphillon.v1` | core: plugins, plugin service, workflows, permissions, version |
| `sapphillon.ai.v1` | LLM models, providers, model search |
| `sapphillon.arcana.v1` | plugin packaging config, plugin manifests |

External proto deps come from the `buf.build/googleapis/googleapis` module (pinned in `buf.lock`), so `google/rpc/status.proto`, `google/protobuf/timestamp.proto`, and `google/protobuf/field_mask.proto` are available without vendoring. To add a new `buf.build` dependency, run `buf dep update`.

## Buf is the toolchain

Buf (`buf.yaml`, v2 config) governs linting, formatting, and code generation.

- **Lint rules in force: `STANDARD` + `COMMENTS`.** The `COMMENTS` rule is the one most likely to bite you — every service, RPC, message, and field must carry a leading comment.
- Breaking-change checks use the `FILE` level (`buf breaking`).

### Verify proto changes (run before committing)

```sh
buf lint proto           # style + comments — CI runs this
buf format proto -d      # formatting diff — CI runs this; any output = failure
```

Fix formatting with `buf format proto -w` (or `make buf_fix_style`). `make buf_check_style` runs both checks.

## Required proto file conventions

These are enforced by `buf lint`, not stylistic preferences — an agent will fail CI without them:

- **License header** at the top of every `.proto`: Apache 2.0, `Copyright 2025 Yuta Takahashi`. Copy it verbatim from `proto/sapphillon/v1/version.proto`.
- `syntax = "proto3";` and an explicit `package sapphillon.<domain>.v1;`.
- Leading comment on **every** service, RPC, message, and field (the `COMMENTS` lint rule).
- Snake_case `.proto` filenames. Services named `XxxService`; RPC request/response messages paired as `XxxRequest` / `XxxResponse`.
- Keep versioned files under `.../v1/` paths; do not edit a released version in place if it breaks wire compatibility.

## Docusaurus docs site (`docs/`)

`docs/` is a Docusaurus 3.9 app (Node >= 20; **run all commands from the `docs/` directory**) that renders the proto API using `docusaurus-protobuffet`.

```sh
cd docs
npm ci
npm run gen-docs   # regenerate proto fixtures + protodocs from ../proto
npm run build      # production build — CI runs this
npm run start      # local dev server
npm run typecheck  # tsc
```

### Critical: generated docs files ARE committed

`docs/fixtures/proto.json` and `docs/protodocs/**/*.mdx` are **generated but checked into git** (they are not gitignored). After changing anything in `proto/`, you must run `npm run gen-docs` and commit the regenerated files — otherwise:

- the website ships stale API docs, and
- CI (`check_docs`) builds against the *committed* fixtures, not your local protos.

In contrast, the standalone HTML from `buf.gen.doc.yaml` (`make docs`) writes to root `gen/`, which **is** gitignored — do not commit `gen/`.

### Two codegen templates — don't confuse them

- `buf.gen.json.yaml` → `docs/fixtures/proto.json` → feeds the Docusaurus site (regenerated by `npm run gen-docs`).
- `buf.gen.doc.yaml` → `gen/docs/index.html` → standalone HTML reference (`make docs`); output is gitignored.

## CI

| Workflow | Triggers | What it does |
|---|---|---|
| `check_protobuf.yml` | changes to `proto/**`, `Makefile` | `buf lint proto` + `buf format proto -d` |
| `check_docs.yml` | PRs to `main` | `npm ci` → `npm run gen-docs` → `npm run build` |
| `deploy_docs.yml` | push to `main` (proto/docs/buf changes) | build + deploy to GitHub Pages |
| `release.yml` | **manual** `workflow_dispatch` only | bump semver tag + create GitHub Release |

GitHub Pages site: `https://sapphillon.github.io/Sapphillon_API/` (base path `/Sapphillon_API/`).

## Environment notes

- The devcontainer (`.devcontainer/devcontainer.json`) is labeled **"Rust"** — a leftover from a template. This is **not** a Rust project; ignore that label.
- The required tools *are* present in the dev container: `buf`, `node` (via nvm), `npm`, and `evans` (a gRPC REPL, used by `make buf_debug`).
- `Makefile` targets: `buf_check_style`, `buf_fix_style`, `buf_debug`, `docs`.
